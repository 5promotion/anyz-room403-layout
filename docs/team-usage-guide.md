# チーム利用ガイドライン

> 複数の社員が共通ディレクトリで Claude Code を利用する際のルール。
> 全員が必ず読んでから運用を開始すること。

---

## 禁止事項

### 1. `active-context.md` の同時編集

このファイルは**単一のセッション状態**を保持する設計。複数人が同時にセッション終了→書き込みを行うと、先に書いた人の内容が消失する。

- 同時作業時は **自分のセッションサマリー（`memory/sessions/`）だけを書く**
- `active-context.md` は**最後にセッションを終了した人が更新**する（または当番制にする）

### 2. 個人情報・認証情報の記載

業務上の機密情報（マーケティング予算・施策・KPI・戦略等）の記載は問題ないが、**個人情報と認証情報**は絶対に記載しない：

**記載禁止（個人情報）:**
- 顧客・取引先の個人名、連絡先、住所
- 社員の給与・評価・人事情報
- マイナンバー、クレジットカード番号等

**記載禁止（認証情報）:**
- APIキー、パスワード、トークン
- `.env` ファイルの内容
- SSHキー、証明書

**記載OK（業務情報）:**
- マーケティング予算・広告費・CPA等のKPI
- キャンペーン施策・戦略・スケジュール
- 競合分析・市場調査データ
- 案件の見積金額・契約条件

認証情報は **1Password 等のシークレット管理ツール**で管理し、ファイル中では「1Password の〇〇を参照」と**ポインタだけ記載**する。

### 3. 他人のセッションファイルの改変・削除

`memory/sessions/` 内の他人のサマリーを勝手に編集・削除しない。  
誤りを見つけた場合は**本人に連絡**する。

### 4. `CLAUDE.md` の無断変更

Layer 1 は全員のセッションに自動注入される。変更は**チームで合意してから**行う。  
個人の好みは `profile/preferences.md` や個人用ファイルに記載する。

### 5. 大量ファイルの一括生成

Claude Code に「ファイルをたくさん作って」と指示すると、ディレクトリが肥大化し Google Drive の同期負荷が増大する。1セッションで作成するファイルは**必要最小限**に留める。

## 注意事項

### 1. Google Drive 同期の競合

Google Drive はファイル競合時に `ファイル名 (競合コピー).md` を生成する場合がある。

- 作業前に Google Drive の**同期が完了していることを確認**する
- 競合コピーが発生したら、内容を比較・統合して競合コピーを削除する
- 特に `decisions.md` と `insights.md` は追記型なので競合しやすい

### 2. セッションファイルの命名規則

`memory/sessions/` 内のファイルは以下の命名規則を厳守する：

```
YYYY-MM-DD_{名前}_{概要}.md
```

**例:**
- `2026-04-08_田中_月次レポート作成.md`
- `2026-04-08_佐藤_新規提案書ドラフト.md`

名前を含めることで、誰の作業記録かが一目で分かる。

### 3. `decisions.md` への追記ルール

意思決定ログは**追記のみ**（既存エントリの編集禁止）。  
各エントリに**決定者名**を含める：

```markdown
## 2026-04-08: 決定タイトル
- **決定者**: 田中
- **決定**: 何を決めたか
- **根拠**: なぜその決定をしたか
```

### 4. 作業前の同期確認

Claude Code でセッションを開始する前に、必ず以下を確認する：

1. Google Drive の同期が完了している（アイコンがチェックマーク）
2. `active-context.md` を読み、他の人が作業中でないか確認
3. 作業中であれば、`active-context.md` の更新を自分は行わない旨を意識する

### 5. Layer 3 ファイルの追加

`.claude/skills/` や `docs/` にファイルを追加する場合：

- ファイル先頭に `> 作成者: 名前 / 作成日: YYYY-MM-DD` を記載する
- 既存スキルと重複しないか事前に確認する
- 汎用的でないもの（個人の作業メモ等）は置かない

## 推奨利用方法

### 1. セッション開始時の定型フロー

Claude Code に以下を最初に伝える：

```
セッション開始。担当: [自分の名前]
```

Claude は CLAUDE.md の指示に従い、Layer 2 を自動的に読み込む。名前を伝えることで、セッションサマリーの命名やログへの記名が正しく行われる。

### 2. セッション終了時の定型フロー

作業が終わったら：

```
セッション終了。担当: [自分の名前]
```

Claude がハンドオフスキルに従い、セッションサマリーの作成と active-context.md の更新を行う。

### 3. `insights.md` への積極的な知見蓄積

業務で得た学びは積極的に `learnings/insights.md` に追記する。  
これが**チーム全体の資産**になる。記載例：

```markdown
### 2026-04-08: 提案書の構成パターン（田中）
- **出典案件**: projects/2026-04_〇〇提案
- **学び**: 課題→解決策→効果の順で書くと承認率が高い
- **次回への適用**: 提案書作成時のテンプレートとして活用
```

### 4. プロジェクト単位のディレクトリ活用

案件ごとに `projects/YYYY-MM_案件名/` を作成し、`plan.md` を正の情報源とする。  
これにより、他の社員が Claude Code で「〇〇案件の現状を教えて」と聞くだけで、即座に状況把握できる。

### 5. スキルファイルで業務を標準化

繰り返し行う業務は `.claude/skills/` にスキルとして定義する：

```markdown
# 月次レポート作成スキル

> 月次レポートを作成する際の手順。

## 実行手順
1. `projects/` から当月の案件をリストアップ
2. 各案件の `plan.md` から進捗を抽出
3. `memory/decisions.md` から当月の重要決定を抽出
4. レポートテンプレートに沿って出力
```

これにより、**誰が作業しても同じ品質**のアウトプットが出せる。

### 6. 困ったときは `decisions.md` と `insights.md` を参照

判断に迷ったら、まず過去の意思決定と知見を確認する。  
Claude Code に「過去に似た判断はあるか？」と聞けば、蓄積された記録から根拠を提示してくれる。

## 運用体制の推奨

| 役割 | 担当 | 責任 |
|------|------|------|
| **CLAUDE.md 管理者** | 1名（固定） | Layer 1 の変更権限。チーム合意後に反映 |
| **ナレッジ整理担当** | 持ち回り（月次） | `insights.md` の重複整理、古いセッションのアーカイブ |
| **リソース更新担当** | 1名（固定） | `resources.md` を月1回以上更新（30日ルール） |

## システムによる防止策

以下は「ルールで禁止」ではなく「仕組みで不可能にする」ための技術的ガードレール。

### 1. Claude Code フック（`.claude/settings.json`）

Claude Code の PreToolUse フックにより、Claude が保護対象ファイルを編集しようとした時点で**自動ブロック**される。

| フック | 対象 | 動作 |
|--------|------|------|
| `protect-files.sh` | `CLAUDE.md`, `.claude/settings.json`, `.claude/hooks/` | Edit/Write を即座にブロック |
| `block-personal-info.sh` | 全ファイルへの書き込み | 電話番号・マイナンバー・カード番号を検出してブロック |

- 設定ファイル: `.claude/settings.json`
- スクリプト: `.claude/hooks/` 配下
- **社員が意識しなくても、Claude 側で自動的に防止される**

### 2. GitHub CODEOWNERS + Branch Protection

GitHub 上で保護対象ファイルの変更に**管理者レビューを必須化**する。

- `.github/CODEOWNERS` で `CLAUDE.md` 等のオーナーを管理者に指定済み
- GitHub リポジトリ設定で以下を有効にする（手動設定が必要）:
  - Settings → Branches → Branch protection rules → `main`
  - `Require a pull request before merging` を有効化
  - `Require review from Code Owners` を有効化

これにより、**管理者の承認なしには main ブランチに反映されない**。

### 3. カスタマイズ方法

**保護対象ファイルを追加する場合:**

`.claude/hooks/protect-files.sh` の `PROTECTED_FILES` 配列に追加:

```bash
PROTECTED_FILES=(
  "CLAUDE.md"
  ".claude/settings.json"
  ".claude/hooks/"
  "profile/resources.md"    # ← 追加例
)
```

**個人情報パターンを追加する場合:**

`.claude/hooks/block-personal-info.sh` にパターンを追記する。

**社内メールドメインを設定する場合:**

`block-personal-info.sh` 内の `@example\.com` を実際の社内ドメインに変更する。

## トラブルシューティング

| 症状 | 原因 | 対処 |
|------|------|------|
| Claude が古い情報を参照する | `active-context.md` が未更新 | 前回作業者にハンドオフを依頼 |
| 同じファイルに競合コピーが発生 | 同時編集 | 内容を手動マージし競合コピーを削除 |
| `resources.md` の警告が出る | 更新日から30日超過 | リソース更新担当が最新情報に更新 |
| Claude が個人の好みを無視する | `preferences.md` 未記入 | 自分の好み・判断基準を記入 |
| セッション履歴が見つからない | 命名規則の不備 | `YYYY-MM-DD_{名前}_{概要}.md` に修正 |
